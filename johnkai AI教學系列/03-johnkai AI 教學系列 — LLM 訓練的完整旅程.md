# 第 2 章：訓練的本質——讓錯誤變小

## 從隨機權重開始

一個剛初始化的 LLM，所有的權重矩陣（W_Q、W_K、W_V、W_O、W_up、W_gate、W_down、W_embed、W_head……）裡面裝的都是**隨機數字**。

具體來說，常見的初始化方法是**Xavier 初始化**或**Kaiming 初始化**——它們不是完全隨機，而是根據矩陣的維度決定隨機數的範圍，讓初始的輸出值不會太大也不會太小。

```
以 Llama 3.1 8B 為例，W_Q 的形狀是 [4096 × 4096]。
Xavier 初始化會把每個元素設為均值=0、標準差=√(2/(4096+4096))≈0.0156 的隨機數。

這意味著：模型一開始完全不知道任何語言知識。
你輸入「什麼是機器學習？」，模型的輸出跟擲骰子沒有兩樣。
```

訓練的目標，就是把這些隨機數字調整成「有用的」數字——讓模型面對任何輸入，都能給出合理的下一個 Token 預測。

---

## 怎麼衡量「錯多少」？——損失函數（Loss Function）

要調整權重，首先要有一個**衡量標準**：模型現在的預測有多糟糕？

Doc2 已經介紹過，模型的最後一步是把隱藏向量投影到詞彙表大小的 Logits，經過 Softmax 變成機率分佈。訓練時，我們知道正確答案是什麼（下一個 Token），所以可以比較模型的預測和正確答案之間的差距。

這個差距的度量，叫**損失函數**（**Loss Function**）。

---

### 損失函數（Loss Function）

**為什麼需要它**

模型輸出的是一個機率分佈（128,256 個詞各自的機率），正確答案是其中一個 Token。你需要一個數字來表達「這個機率分佈離正確答案有多遠」。這個數字就是 Loss。

Loss 越大，預測越糟；Loss 越小，預測越準。訓練的目標就是讓 Loss 盡可能小。

**回到故事**

我們的模型剛初始化，所有權重是隨機的。你輸入「今天天氣」，正確的下一個 Token 應該是「很」。但模型輸出的機率分佈是隨機的——「很」這個 Token 的機率可能只有 1/128256 ≈ 0.00078%。

這個預測非常糟糕。Loss 應該很大。

**往下延伸——為什麼選 Cross-Entropy？**

衡量「一個機率分佈離正確答案有多遠」有很多方法。LLM 訓練幾乎都用**交叉熵損失**（**Cross-Entropy Loss**）。

---

### 交叉熵損失（Cross-Entropy Loss, CE Loss）

**為什麼需要它**

我們需要一個損失函數滿足三個條件：
1. 當模型給正確答案的機率越高，Loss 越小
2. 當模型給正確答案的機率趨近於 0，Loss 趨近於無窮大——嚴厲懲罰「完全猜錯」
3. 它的梯度（導數）計算簡單，適合反向傳播

Cross-Entropy 完美滿足這三個條件。

**它是什麼**

假設詞彙表大小是 V（比如 128,256）。模型對位置 t 的預測是一個機率向量 p = [p₁, p₂, ..., p_V]，正確答案是第 c 個 Token（其中 c 是正確 Token 的索引）。

Cross-Entropy Loss 的公式：

```
L = -log(p_c)

其中 p_c 是模型給正確答案的機率。
```

就這麼簡單。**取正確答案的機率，取對數，加負號。**

**為什麼用 -log？**

用一個數值例子感受：

```
情境 1：模型很確定，p_c = 0.9（90% 機率給了正確答案）
  L = -log(0.9) = 0.105
  Loss 很小 ✓

情境 2：模型亂猜，p_c = 0.01（只有 1% 機率給了正確答案）
  L = -log(0.01) = 4.605
  Loss 很大 ✓

情境 3：模型完全猜錯，p_c = 0.0001
  L = -log(0.0001) = 9.210
  Loss 非常大 ✓

情境 4：完美預測，p_c = 1.0
  L = -log(1.0) = 0
  Loss 為零 ✓
```

-log 函數的性質完美對應了我們的直覺：

```
p_c:  1.0 ──→ 0.5 ──→ 0.1 ──→ 0.01 ──→ 0.001 ──→ 0
Loss:  0  ──→ 0.69 ──→ 2.30 ──→ 4.61 ──→ 6.91 ──→ ∞

越接近 0，Loss 越陡峭地增長——嚴厲懲罰「完全猜錯」。
越接近 1，Loss 越平緩——不會因為微小的改進就劇烈波動。
```

**Cross-Entropy 的梯度**

Cross-Entropy 搭配 Softmax 有一個非常優美的梯度公式。回憶 Doc2，Softmax 把 Logits 轉成機率：

```
p_i = exp(z_i) / Σ_j exp(z_j)
```

Cross-Entropy Loss 對 Logits z_i 的梯度：

```
∂L/∂z_i = p_i - y_i

其中 y_i 是 one-hot 標籤：正確答案位置=1，其他位置=0。
```

也就是說：**梯度 = 預測機率 - 正確答案**。

```
對於正確答案（y_c = 1）：∂L/∂z_c = p_c - 1
  如果 p_c = 0.9 → 梯度 = -0.1（小修正）
  如果 p_c = 0.01 → 梯度 = -0.99（大修正）

對於錯誤答案（y_i = 0）：∂L/∂z_i = p_i
  如果 p_i = 0.5 → 梯度 = 0.5（壓低這個錯誤選項的分數）
  如果 p_i = 0.001 → 梯度 = 0.001（幾乎不動）
```

這個梯度性質極好：
- **預測越錯，修正力道越大**（自動加速學習）
- **預測越對，修正力道越小**（不會過度調整）
- **計算極簡**：只需要做一次減法

這就是為什麼 Cross-Entropy + Softmax 是 LLM 訓練的黃金組合。

**搭配判讀**

Cross-Entropy Loss 的絕對值本身不好直接比較不同模型——因為詞彙表大小不同，隨機猜的 Loss 基準就不同：

```
隨機猜的 Loss = -log(1/V)

Llama 3.1（V = 128,256）：-log(1/128256) = 11.76
一個小模型（V = 32,000）：-log(1/32000) = 10.37
```

所以 Loss 從 11.76 降到 3.0 表示「從隨機猜進步到每個位置平均考慮 e³ ≈ 20 個合理選項」——這是指數級的進步。

另一個常用的指標是**困惑度**（**Perplexity, PPL**）：

```
PPL = exp(Loss)

Loss = 3.0 → PPL = e³ ≈ 20
Loss = 2.0 → PPL = e² ≈ 7.4
Loss = 1.0 → PPL = e¹ ≈ 2.7
```

PPL 的直覺意義：模型在每個位置平均在幾個 Token 之間猶豫。PPL = 20 表示平均在 20 個 Token 之間猶豫；PPL = 2.7 表示幾乎確定是哪一個。

> 訓練的本質就是最小化 Cross-Entropy Loss——讓模型給正確 Token 的機率盡可能高。Loss 的梯度「預測 - 正確答案」簡單而精準，是驅動整個訓練過程的核心引擎。

---

## 每個位置都是一個訓練樣本

一個關鍵但容易忽略的事實：**一句話裡的每個位置都提供一個訓練信號**。

```
輸入序列：「今 天 天 氣 很 好」（6 個 Token）

位置 1：輸入「今」，正確答案「天」 → 算一次 Loss
位置 2：輸入「今 天」，正確答案「天」 → 算一次 Loss
位置 3：輸入「今 天 天」，正確答案「氣」 → 算一次 Loss
位置 4：輸入「今 天 天 氣」，正確答案「很」 → 算一次 Loss
位置 5：輸入「今 天 天 氣 很」，正確答案「好」 → 算一次 Loss

一句 6 個 Token 的話，提供 5 個訓練樣本。
```

但其實 Transformer 用了 Causal Mask（Doc1 已介紹），每個位置只能看到自己和前面的 Token。所以**一次 Forward Pass 就能同時計算所有位置的 Loss**：

```
模型一次接收 [今, 天, 天, 氣, 很, 好]
                ↓    ↓    ↓    ↓    ↓
Causal Mask 讓每個位置只看到前面的 Token
                ↓    ↓    ↓    ↓    ↓
同時預測      [天,  天,  氣,  很,  好,  <下一個>]
                ↓    ↓    ↓    ↓    ↓
同時算 5 個 Cross-Entropy Loss
                ↓
取平均 → 這句話的總 Loss

L_total = (1/5) × (L₁ + L₂ + L₃ + L₄ + L₅)
```

這就是為什麼 Transformer 的訓練效率遠高於 RNN——RNN 必須逐步計算，Transformer 一次並行算完所有位置。

---

### 下一個 Token 預測（Next Token Prediction, NTP）

**為什麼選這個目標**

LLM 的 Pre-training 目標是 **Next Token Prediction（NTP）**：給定前面的 Token 序列，預測下一個 Token。

為什麼不教模型「理解語言」或「回答問題」？因為：

1. **無需人工標注**：網路上有幾兆 Token 的文本。每個文本天然地提供了無數的（前文 → 下一個 Token）訓練樣本，不需要任何人工標注。
2. **蘊含了語言的一切**：要準確預測下一個 Token，模型必須理解語法、語義、邏輯、事實知識、甚至推理能力。預測「牛頓發現了萬有___」的下一個 Token，模型必須知道物理學史。
3. **規模可擴展**：訓練資料越多、模型越大，NTP 的準確率就持續提升（Scaling Law）。

**所以 NTP 不是在教模型「續寫」——NTP 是在用「續寫」這個代理任務，迫使模型學會語言的深層結構。**

**回到故事**

我們的隨機初始化模型現在有了目標：對每個位置預測下一個 Token，用 Cross-Entropy Loss 衡量預測的好壞。接下來的問題是：**怎麼根據 Loss 調整權重？**

Doc1 已經教過反向傳播（Backpropagation）的概念——從 Loss 往回追，用鏈式法則算出每個權重對 Loss 的貢獻（梯度）。現在我們要更深入：**拿到梯度之後，具體怎麼更新權重？**

> NTP 是 LLM Pre-training 的核心目標——用「預測下一個 Token」這個看似簡單的任務，從海量文本中學到語言的所有知識。Cross-Entropy Loss 是衡量預測好壞的標尺，而梯度是更新權重的方向盤。

# 第 3 章：怎麼讓錯誤變小？——優化器

## 從梯度到更新

Doc1 介紹過梯度下降的基本公式：

```
w_new = w_old - η × ∂L/∂w

η = 學習率（Learning Rate）
∂L/∂w = Loss 對權重 w 的梯度
```

每個權重都按照這個公式更新：**往梯度的反方向走一步，步長由學習率控制**。

這個最基本的版本叫 **SGD（Stochastic Gradient Descent，隨機梯度下降）**。

但 SGD 對 LLM 來說有嚴重問題。

---

### SGD 的問題

```
問題 1：所有權重用同一個學習率
  有些權重的梯度很大（經常被更新），有些很小（很少被更新）。
  同一個學習率對前者太大（震盪），對後者太小（學不動）。

問題 2：梯度很嘈雜
  每次只用一個 batch 的資料算梯度（而不是全部資料）。
  單次梯度可能方向偏差很大，導致更新忽左忽右。

問題 3：容易卡在鞍點
  高維空間（LLM 有幾十億維）中，Loss 地形充滿鞍點。
  SGD 在鞍點附近梯度接近 0，更新幾乎停滯。
```

這三個問題催生了更聰明的優化器。

---

### Adam（Adaptive Moment Estimation）

**為什麼需要它**

Adam 同時解決了 SGD 的三個問題：
- 為每個權重自動調整學習率（自適應）
- 用歷史梯度的平均值降噪（動量）
- 用歷史梯度的平方平均值衡量波動（二階矩）

**它是什麼**

Adam 為每個權重維護兩個額外的狀態：

```
m = 一階矩（梯度的指數移動平均）→ 「方向」
v = 二階矩（梯度平方的指數移動平均）→ 「波動程度」
```

每一步的更新過程：

```
第 1 步：計算梯度
  g_t = ∂L/∂w（反向傳播得到）

第 2 步：更新一階矩（帶動量的方向平均）
  m_t = β₁ × m_{t-1} + (1-β₁) × g_t
  β₁ 通常 = 0.9
  含義：新的方向 = 90% 的歷史方向 + 10% 的當前梯度
  效果：平滑掉單次梯度的噪音

第 3 步：更新二階矩（波動程度的平均）
  v_t = β₂ × v_{t-1} + (1-β₂) × g_t²
  β₂ 通常 = 0.999
  含義：追蹤這個權重的梯度歷史波動有多大

第 4 步：偏差校正（前幾步的修正）
  m̂_t = m_t / (1 - β₁ᵗ)
  v̂_t = v_t / (1 - β₂ᵗ)
  含義：訓練剛開始時 m 和 v 都初始化為 0，
       不校正的話前幾步的估計值會偏低

第 5 步：更新權重
  w_new = w_old - η × m̂_t / (√v̂_t + ε)
  ε = 1e-8（防止除以零）
```

**關鍵直覺：** `m̂_t / √v̂_t` 這個除法就是自適應學習率的核心——

```
如果某個權重的梯度一直很大且方向一致：
  m̂ 大，v̂ 也大 → m̂/√v̂ ≈ 穩定的中等值 → 穩步更新

如果某個權重的梯度很小（稀疏特徵）：
  m̂ 小，v̂ 更小 → m̂/√v̂ 反而較大 → 放大更新步長

如果某個權重的梯度忽大忽小（噪音大）：
  m̂ 被平滑掉大部分噪音 → v̂ 反映高波動 → m̂/√v̂ 很小 → 謹慎更新
```

每個權重自動得到了適合自己的有效學習率。

### AdamW（Adam with Decoupled Weight Decay）

**為什麼需要它**

Adam 有一個微妙的問題：**Weight Decay（權重衰減）被混進了自適應學習率裡。**

Weight Decay 是一種正則化技術——每一步把所有權重稍微推向 0，防止權重變得太大（過擬合）：

```
原始 Weight Decay：w_new = w_old × (1 - λ) - η × gradient
λ 是衰減係數，通常 = 0.01 或 0.1
```

但在 Adam 裡，如果把 Weight Decay 當成 L2 正則化加進 Loss（L = L_original + λ × ||w||²），那 Weight Decay 的梯度也會被 Adam 的 v 項（二階矩）縮放。結果是：**梯度波動大的權重，Weight Decay 反而變弱了**——跟我們想要的均勻衰減相反。

**AdamW 的解法**

AdamW 把 Weight Decay 從 Adam 的梯度更新中分離出來：

```
Adam（Weight Decay 耦合在梯度裡）：
  g_t = ∂L/∂w + λ × w    ← Weight Decay 混進梯度
  m_t = β₁ × m_{t-1} + (1-β₁) × g_t
  v_t = β₂ × v_{t-1} + (1-β₂) × g_t²
  w_new = w_old - η × m̂_t / (√v̂_t + ε)

AdamW（Weight Decay 獨立）：
  g_t = ∂L/∂w             ← 純粹的梯度，不含 Weight Decay
  m_t = β₁ × m_{t-1} + (1-β₁) × g_t
  v_t = β₂ × v_{t-1} + (1-β₂) × g_t²
  w_new = w_old - η × m̂_t / (√v̂_t + ε) - η × λ × w_old
                                            ↑
                                    Weight Decay 獨立作用
```

差異就在最後一項：AdamW 直接對權重本身做衰減，不經過 Adam 的自適應縮放。

**為什麼這很重要？**

```
AdamW 的效果：
  所有權重被均勻地推向 0 → 正則化效果穩定
  梯度更新由 Adam 自適應控制 → 學習效率不受影響
  兩個機制互不干擾 → 超參數（η 和 λ）可以獨立調整

這就是為什麼 2026 年幾乎所有 LLM 的 Pre-training 都用 AdamW。
```

**Adam 和 AdamW 的記憶體成本**

Adam/AdamW 為每個權重維護 m 和 v 兩個狀態，加上權重本身，總共需要 3 份權重大小的記憶體：

```
以 Llama 3.1 8B（80 億參數）為例：
  權重本身（FP16）：8B × 2 bytes = 16 GB
  m 狀態（FP32）：  8B × 4 bytes = 32 GB
  v 狀態（FP32）：  8B × 4 bytes = 32 GB
  梯度（FP16）：    8B × 2 bytes = 16 GB
  ──────────────────────────────────
  總計：約 96 GB

一張 A100 80GB 都裝不下——這就是為什麼訓練需要多張 GPU。
```

> 優化器是訓練的引擎——從 SGD 的「所有權重同速」，到 Adam 的「每個權重自適應」，到 AdamW 的「解耦 Weight Decay」。AdamW 是目前 LLM 訓練的事實標準，但它的代價是每個參數需要 3 倍記憶體。

---

## 學習率排程（Learning Rate Schedule）

學習率 η 不是一個固定值——它在訓練過程中按照預設的排程變化。

### 為什麼需要排程

```
如果學習率太大（全程 η = 1e-3）：
  訓練前期：進步很快 ✓
  訓練後期：在最優解附近震盪，無法收斂 ✗

如果學習率太小（全程 η = 1e-5）：
  訓練前期：進步極慢，浪費計算資源 ✗
  訓練後期：可以精細調整 ✓

所以最佳策略是：前期大、後期小。
```

### Warmup + Cosine Decay

目前 LLM Pre-training 最常用的排程：

```
學習率
  ↑
η_max ─────────────╮
  │               ╱  ╲
  │              ╱    ╲  Cosine Decay
  │             ╱      ╲
  │            ╱        ╲
  │           ╱          ╲
  │   Warmup╱            ╲
  │        ╱              ╲
η_min ───╱                 ╲──────
  └──────┬──────┬──────────┬────→ Training Steps
         0   T_warmup    T_total

三個階段：
  1. Warmup（前 1-10% 的步數）：
     學習率從 η_min 線性增加到 η_max
     原因：訓練剛開始時梯度不穩定，大學習率會導致發散

  2. Cosine Decay（中間大部分）：
     學習率按餘弦曲線從 η_max 平滑衰減到 η_min
     公式：η_t = η_min + 0.5 × (η_max - η_min) × (1 + cos(π × t/T))

  3. 結束：學習率降至 η_min（通常是 η_max 的 1/10 或更小）
```

**典型的超參數配置**（以 Llama 3.1 8B 為例）：

```
  η_max = 3e-4
  η_min = 3e-5（η_max 的 1/10）
  T_warmup = 2,000 步
  T_total = ~1,000,000 步（15T tokens / batch_size）
  β₁ = 0.9
  β₂ = 0.95
  Weight Decay λ = 0.1
```

**最新趨勢（2025-2026）**

近期研究發現，Cosine Decay 不一定是唯一的好選擇：

```
Warmup-Stable-Decay（WSD）：
  把衰減推遲到訓練的最後 10-20%
  大部分時間保持穩定的高學習率
  優勢：如果提前停止訓練，模型品質比 Cosine 好

Linear Decay：
  簡單地從 η_max 線性衰減到 0
  2025 年研究顯示效果接近 Cosine，而且更容易調參
```

但 Cosine Decay 仍然是最廣泛使用的排程，因為大量的經驗和基準都建立在它之上。

> 學習率排程是「先快後慢」的訓練策略——Warmup 防止起步爆炸，Cosine Decay 讓收斂越來越精細。AdamW + Warmup + Cosine Decay 構成了 LLM Pre-training 的標準優化配方。

# 第 4 章：用什麼資料訓練？——資料工程

模型有了、損失函數有了、優化器有了。接下來的問題是：**用什麼資料餵給模型？**

Pre-training 的資料來源是網路上的海量文本。但不是把網頁爬下來就能直接用——原始資料充滿噪音、重複、低品質內容。資料工程的工作就是把原始資料變成乾淨的訓練資料。

---

## 資料管線（Data Pipeline）

一個典型的 Pre-training 資料管線：

```
原始網頁資料（幾百 TB）
    │
    ▼
① 語言識別（Language Identification）
    過濾非目標語言
    │
    ▼
② 品質過濾（Quality Filtering）
    移除低品質內容
    │
    ▼
③ 去重（Deduplication）
    移除重複和近似重複的文本
    │
    ▼
④ 有害內容過濾（Toxicity Filtering）
    移除仇恨言論、違法內容等
    │
    ▼
⑤ PII 過濾（Personal Information）
    移除個人隱私資訊
    │
    ▼
⑥ 分詞（Tokenization）
    文本 → Token ID 序列
    │
    ▼
⑦ 序列打包（Sequence Packing）
    拼接成固定長度的訓練樣本
    │
    ▼
乾淨的訓練資料（幾 TB Token）
```

---

### 品質過濾

**為什麼需要它**

網路上的文本大多數是低品質的——導航選單、廣告文字、亂碼、機器生成的垃圾內容。如果不過濾，模型會花大量計算學習這些無用的模式。

**常見的過濾規則**：

```
啟發式過濾（基於規則）：
  - 移除太短的文件（< 200 字元）
  - 移除重複比例太高的文件（同一句話重複 5 次以上）
  - 移除特殊字元比例太高的文件（>30% 非文字字元）
  - 移除「單詞」平均長度異常的文件

模型驅動過濾（2024-2025 趨勢）：
  - 用一個分類器模型（如 fastText）判斷文本是否「像教科書」
  - FineWeb-Edu 資料集就是這樣建立的——用模型給每段文字打「教育品質分數」
  - 效果顯著：同等資料量下，品質過濾後的資料訓練出的模型明顯更好
```

---

### 去重（Deduplication）

**為什麼需要它**

網路上大量的文本是重複的——同一篇文章被轉載幾十次，版權聲明和免責條款在每個頁面都出現。重複的資料會導致：

```
問題 1：浪費計算——同一段文字被模型看了幾十遍，沒有新的學習信號
問題 2：記憶化（Memorization）——模型背下特定文本，而不是學到通用模式
問題 3：評估污染——如果測試集的內容出現在訓練資料中，評估結果就不可信
```

**常見的去重方法**：

```
精確去重（Exact Deduplication）：
  計算每段文字的 hash 值
  完全相同的 hash → 刪除重複項
  快但粗糙——改一個字就躲過了

近似去重（Near-Duplicate，MinHash LSH）：
  把文本切成 n-gram（連續 n 個 Token 的片段）
  用多個 hash 函數估計兩段文字的相似度（Jaccard 相似度）
  相似度超過閾值（如 80%）→ 視為近似重複，刪除

  MinHash LSH（Locality-Sensitive Hashing）可以在幾十億文件中
  高效找到近似重複的對——這是 LLM 資料處理的核心技術之一

軟去重（SoftDedup，2024-2025）：
  不直接刪除重複，而是降低重複內容的權重
  保留資訊的同時減少重複訓練
```

---

### 序列打包（Sequence Packing）

**為什麼需要它**

Transformer 的輸入是固定長度的序列（比如 4,096 或 8,192 Token）。但網路上的文件長短不一——有的 50 Token，有的 50,000 Token。

```
序列打包的做法：
  1. 把不同文件拼接在一起，中間用特殊 Token（如 <EOS>）分隔
  2. 切成固定長度的 chunk
  3. 每個 chunk 就是一個訓練樣本

  文件 A（200 Token）+ <EOS> + 文件 B（3,000 Token）+ <EOS> + 文件 C（896 Token）
  = 4,097 Token → 切成 [4,096] 的訓練樣本 + 1 Token 進下一個樣本

  注意力 Mask 會確保不同文件之間不互相注意——
  文件 A 的 Token 看不到文件 B 的 Token，即使它們在同一個序列裡。
```

---

### 資料配比（Data Mix）

Pre-training 不只用一種資料。典型的配比：

```
以 Llama 3.1 為例（15T Token 的訓練資料）：

  網頁資料：~50%（CommonCrawl 過濾後）
  程式碼：~17%（GitHub 等）
  書籍/學術論文：~8%
  維基百科/百科全書：~5%
  數學/科學：~5%
  多語言資料：~10%
  其他（對話、問答等）：~5%

配比的調整是一門藝術——
  程式碼比例高 → 模型推理能力更強
  數學比例高 → 模型計算能力更強
  維基百科比例高 → 模型事實知識更紮實
```

### 資料品質 > 資料量

2024-2025 年的一個重要發現：

```
FineWeb-Edu 實驗：
  用模型驅動過濾，從 15T Token 中篩選出 1.3T 的高品質「教育級」文本。
  用 1.3T 高品質資料訓練的模型，在多個基準上超過用 15T 原始資料訓練的模型。

結論：1T 乾淨資料 > 10T 髒資料。
```

這個發現也解釋了一個趨勢：最新的模型雖然參數量不一定更大，但訓練資料的品質持續提升。

> 資料工程是 Pre-training 的「地基」——品質過濾、去重、配比決定了模型能學到什麼。「Garbage in, garbage out」在 LLM 訓練中體現得淋漓盡致：1T 乾淨資料的價值超過 10T 未清洗的資料。

---

# 第 5 章：要訓練多大、多久？——Scaling Law

模型、損失函數、優化器、資料都準備好了。但一個關鍵的工程決策還沒做：

**模型要多大？訓練多久？**

這兩個問題直接決定了訓練的成本——而成本是以千萬美元計的。

---

## Scaling Law 的核心發現

2020 年，OpenAI 的研究團隊（Kaplan 等人）發表了一個重要發現：

```
LLM 的 Loss 與三個因素呈冪律關係：

  L(N) ∝ N^(-α)    模型參數量 N 越大，Loss 越低
  L(D) ∝ D^(-β)    訓練資料量 D 越大，Loss 越低
  L(C) ∝ C^(-γ)    計算量 C 越大，Loss 越低

  α ≈ 0.076, β ≈ 0.095, γ ≈ 0.050

冪律（Power Law）的含義：
  增加 10 倍的參數/資料/計算 → Loss 下降一個固定的量
  這種關係在好幾個數量級上都成立
```

**「好幾個數量級」有多穩定？**

```
從 1000 萬參數到 1000 億參數（4 個數量級），
Loss 與 log(N) 幾乎完美線性。

這意味著：你可以用小模型的實驗結果，預測大模型的表現。
這是 Scaling Law 最重要的實際價值——不需要真的訓練一個 100B 模型
就能預估它的性能。
```

---

## Chinchilla 定律（2022）

Kaplan 的 Scaling Law 告訴你「越大越好」，但沒告訴你**在固定計算預算下，模型和資料怎麼分配最好**。

2022 年，DeepMind 的 Chinchilla 論文回答了這個問題：

### 計算最優的分配

```
給定固定計算量 C，Loss 最小的分配方式是：

  N_opt ∝ C^0.5    最優模型大小與計算量的平方根成正比
  D_opt ∝ C^0.5    最優資料量也與計算量的平方根成正比

簡化版本：
  D_opt ≈ 20 × N_opt

含義：每增加一倍的模型參數，訓練資料量也應該增加一倍。
```

**Chinchilla 的衝擊**

```
在 Chinchilla 之前，業界的做法是「模型越大越好，資料差不多就行」：
  GPT-3 (175B 參數) 只用 300B Token 訓練 → D/N ≈ 1.7

Chinchilla 說最優比例是 D/N ≈ 20。
所以 GPT-3 的資料量少了 10 倍以上！

DeepMind 用同樣的計算量訓練了 Chinchilla (70B 參數)：
  70B 參數 × 20 = 1.4T Token
  表現全面超越 GPT-3 (175B 參數, 300B Token)

結論：用更小的模型 + 更多的資料，比用更大的模型 + 更少的資料效果更好。
```

---

## 超越 Chinchilla：過度訓練（Over-training）

Chinchilla 定律講的是「計算最優」——**如果你只在乎訓練成本**。但實務中還有另一個成本：**推理成本**。

```
訓練成本：一次性的。訓練完就結束。
推理成本：持續的。每次有人使用模型，都要花 GPU 計算。

一個 70B 模型的推理成本 >> 一個 7B 模型的推理成本。
如果你的模型要服務幾百萬用戶，推理成本遠超訓練成本。
```

所以 2024-2025 年出現了一個趨勢：**有意地「過度訓練」小模型**——用遠超 Chinchilla 最優比例的資料量訓練更小的模型：

```
Chinchilla 最優：D/N ≈ 20
  7B 模型應該用 140B Token

實際做法：
  Llama 3.1 8B：15T Token → D/N ≈ 1,875（超 Chinchilla 93 倍）
  Qwen3-0.6B：36T Token → D/N = 60,000（超 Chinchilla 3,000 倍）

為什麼？
  訓練成本是一次性的，多花幾倍的訓練費用，
  換來一個更小但更聰明的模型，
  在推理時每次調用都省錢。

  而且研究發現：模型品質在 D/N >> 20 時仍然持續提升，
  只是提升的速率比 Chinchilla 最優區間慢。
```

---

## 訓練成本的估算

```
計算量的單位是 FLOPs（浮點運算次數）。

經驗公式：C ≈ 6 × N × D
  C = 總 FLOPs
  N = 參數量
  D = 訓練 Token 數
  6 = 每個 Token 每個參數大約需要 6 次浮點運算
     （Forward 2 次 + Backward 4 次，乘加各算一次）

以 Llama 3.1 8B 為例：
  C = 6 × 8B × 15T = 7.2 × 10²³ FLOPs

一張 A100 GPU：
  FP16 算力 = 312 TFLOPS = 3.12 × 10¹⁴ FLOPs/秒
  利用率 ≈ 40%（通訊、等待等開銷）
  有效算力 ≈ 1.25 × 10¹⁴ FLOPs/秒

所需 GPU 時數：
  7.2 × 10²³ / 1.25 × 10¹⁴ = 5.76 × 10⁹ 秒 ≈ 184 GPU-年

如果用 2,048 張 A100：
  訓練時間 ≈ 184 / 2048 × 365 ≈ 33 天

成本（以雲端租用 A100 $2/GPU·小時估算）：
  184 年 × 8760 小時 × $2 ≈ $3.2M（約 320 萬美元）

更大的模型（70B）：
  C = 6 × 70B × 15T = 6.3 × 10²⁴ FLOPs
  用 16,000 張 H100 ≈ 3-4 個月
  成本 ≈ $30-50M（3000-5000 萬美元）
```

> Scaling Law 揭示了 LLM 能力與規模的冪律關係——Chinchilla 定義了計算最優的模型/資料比例（D/N ≈ 20），但實務上「過度訓練」小模型是更經濟的選擇。訓練一個 8B 模型要 ~$3M，70B 要 ~$30M，這就是為什麼只有少數公司能從頭訓練 LLM。

# 第 6 章：Pre-training 的完整流程

前面幾章分別介紹了 Loss、優化器、資料、和 Scaling Law。現在把它們組裝起來，走一遍完整的 Pre-training 流程。

---

## 一步訓練（One Training Step）

```
                      ┌──────────────────────────────────────────┐
                      │              一步訓練                     │
                      │                                          │
   訓練資料           │  ① Forward Pass（前向傳播）              │
   [batch_size ×      │     輸入 Token → Embedding → L 層       │
    seq_len]          │     Transformer Block → LM Head →       │
        │             │     Logits [batch × seq_len × V]        │
        ▼             │                                          │
   ┌────────────┐     │  ② Loss 計算                            │
   │ 一個 Batch │────►│     Cross-Entropy：                     │
   │ 的 Token   │     │     L = -(1/T) Σ log(p_correct)        │
   └────────────┘     │                                          │
                      │  ③ Backward Pass（反向傳播）             │
                      │     從 Loss 往回算每個權重的梯度         │
                      │     ∂L/∂w for all parameters             │
                      │                                          │
                      │  ④ 優化器更新                            │
                      │     AdamW 用梯度更新權重                 │
                      │     更新 m, v 狀態                       │
                      │     應用 Weight Decay                    │
                      │     更新學習率（按排程）                 │
                      │                                          │
                      │  ⑤ 梯度歸零                              │
                      │     清除梯度，準備下一步                 │
                      └──────────────────────────────────────────┘
                                        │
                                        ▼
                                  回到 ①，下一個 batch
```

### Batch 和 Batch Size

**為什麼不用全部資料一起算？**

```
全部資料：15T Token → 記憶體完全放不下
單個樣本：梯度噪音太大，更新方向不穩定

折衷：每次用一個 Batch（批次）的資料算梯度
  Batch 越大 → 梯度越準確，但記憶體需求越大
  Batch 越小 → 記憶體省，但梯度噪音大
```

LLM Pre-training 的 Batch Size 通常非常大：

```
Llama 3.1 8B：
  Batch Size = 4M Token（4,194,304 Token / 步）
  = 1,024 個序列 × 4,096 Token/序列

  15T Token / 4M Token/步 ≈ 3,750,000 步

  也就是說：模型的權重總共被更新了約 375 萬次。
  每次更新都用了 400 萬個 Token 的梯度信號。
```

### 梯度累積（Gradient Accumulation）

一個 4M Token 的 batch 放不進一台機器的記憶體。解決方案：

```
梯度累積：
  把一個大 Batch 拆成多個 Micro-batch
  每個 Micro-batch 做 Forward + Backward，累積梯度
  累積完所有 Micro-batch 後，做一次優化器更新

  例如：Batch Size = 4M Token
  Micro-batch = 4K Token（1 個序列）
  需要累積 1,024 次

  實際上會搭配資料並行（多張 GPU 各算一個 Micro-batch），
  所以每張 GPU 只需要累積一小部分。
```

---

## 訓練過程中的監控

```
Loss 曲線（正常情況）：
  Loss
   ↑
11 ├─╲
   │  ╲
 8 ├   ╲
   │    ╲
 5 ├     ╲╲
   │       ╲╲
 3 ├         ╲──────╲
   │                  ╲───────────
 2 ├                              ──────
   └──────┬──────┬──────┬──────┬──────→ Steps
          0    100K   200K   500K    1M

特徵：
  - 開始時 Loss ≈ 11.76（隨機猜的水平，log(128256)）
  - 前幾萬步下降最快
  - 中間逐漸放緩
  - 後期趨於平穩
```

### Loss Spike（損失突刺）

訓練過程中偶爾會出現 Loss 突然飆升的現象：

```
Loss
  ↑
  │           ╱╲
  │          ╱  ╲ ← Loss Spike
  │         ╱    ╲
3 ├────────╱      ╲──────────
  │
  └──────────────────────────→ Steps

常見原因：
  - 遇到了異常的訓練資料（極端分佈）
  - 學習率太高
  - 數值精度問題（FP16 溢位）

處理方式：
  - 如果 Loss 自行恢復 → 通常可以忽略
  - 如果 Loss 不恢復 → 回滾到 spike 前的 checkpoint，
    跳過問題資料，重新訓練
```

## 混合精度訓練（Mixed Precision Training）

前面提到 AdamW 需要大量記憶體。混合精度訓練是降低記憶體的關鍵技術：

```
核心思想：
  Forward Pass 和 Backward Pass 用 FP16 或 BF16（半精度）
  權重的主副本和優化器狀態用 FP32（全精度）

為什麼不全用 FP16？
  FP16 的範圍：±65,504，精度 ≈ 3-4 位有效數字
  梯度可能非常小（1e-7），FP16 會下溢為 0
  權重更新 w + η×g，如果 η×g 太小，FP16 的精度不夠區分

混合精度的做法：
  1. 保存 FP32 的主權重（Master Weights）
  2. Forward：把主權重轉成 FP16/BF16，算 Forward
  3. Backward：用 FP16/BF16 算梯度
  4. Update：把 FP16 梯度傳回 FP32，在 FP32 精度下更新主權重
  5. 下一步再把更新後的 FP32 權重轉成 FP16 去算 Forward

BF16（Brain Float 16）vs FP16：
  FP16：精度高（10 bit 尾數），但範圍小（±65,504）→ 容易溢位
  BF16：範圍大（8 bit 指數，跟 FP32 一樣），但精度低（7 bit 尾數）
  LLM 訓練更偏好 BF16，因為範圍比精度重要——梯度值跨越很大的量級

記憶體節省：
  Forward + Backward 的 Activation 用 BF16 → 省一半記憶體
  但主權重和優化器仍然是 FP32 → 仍然需要大量記憶體
```

> Pre-training 就是不斷重複「Forward → Loss → Backward → Update」的循環。一個 8B 模型用 15T Token 訓練，意味著這個循環重複約 375 萬次。混合精度和梯度累積讓這個過程在有限的 GPU 記憶體中成為可能。

# 第 7 章：從「續寫機器」到「對話助手」——SFT

## Pre-training 之後的問題

經過幾個月的 Pre-training，模型學會了一件事：**預測下一個 Token**。

這意味著它是一個非常強大的**文本續寫器**。你給它一段文字，它會自然地接著寫下去：

```
輸入：「機器學習是一種」
模型輸出：「人工智慧的子領域，它使用統計方法讓電腦從資料中學習，
而不需要明確程式設計。機器學習的應用包括...」

看起來很好？但問題是：
```

```
輸入：「什麼是機器學習？」
模型輸出：「什麼是深度學習？什麼是自然語言處理？
這些問題的答案可以在《人工智慧導論》中找到。
第一章介紹了機器學習的基本概念...」

它在「續寫」——續寫一份「可能在網路上找到的、包含這句話的文件」。
它沒有在「回答你的問題」。
```

更糟的情況：

```
輸入：「幫我寫一首關於春天的詩」
模型輸出：「，然後我需要把它交給老師。但是我不知道怎麼寫詩，
所以我在網上搜索了『春天的詩』，找到了以下結果...」

它在續寫一段「某個學生在網路上發的帖子」。
```

**問題的根源**：Pre-training 的目標是 NTP（預測下一個 Token），不是「回答問題」或「服從指令」。模型學到的是「網路上的文本長什麼樣」，不是「怎麼當一個助手」。

---

### 監督式微調（Supervised Fine-Tuning, SFT）

**為什麼需要它**

Pre-training 教會模型「語言知識」，但沒有教它「怎麼回答問題」。SFT 的目標是：**教模型「指令 → 回答」的格式**。

**它是什麼**

SFT 用一組精心準備的「指令-回答」對來繼續訓練模型。訓練目標仍然是 NTP（預測下一個 Token），但訓練資料的格式變了。

**SFT 的訓練資料格式**

```
Pre-training 的資料：
  純文本，沒有結構
  「機器學習是人工智慧的子領域。它使用統計方法...」

SFT 的資料：
  結構化的對話，用特殊 Token 標記角色

  <|begin_of_text|>
  <|start_header_id|>system<|end_header_id|>
  你是一個有幫助的助手。
  <|eot_id|>
  <|start_header_id|>user<|end_header_id|>
  什麼是機器學習？
  <|eot_id|>
  <|start_header_id|>assistant<|end_header_id|>
  機器學習是人工智慧的一個分支，它讓電腦能夠從資料中學習，
  而不需要被明確地編程。通過分析大量的訓練資料，
  機器學習算法可以識別模式並做出預測或決策。
  <|eot_id|>
```

**SFT 的訓練方式**

```
關鍵細節：Loss 只在 assistant 回答的部分計算

  <system>...<user>什麼是機器學習？<assistant>機器學習是...
  ──────── 不算 Loss ────────────    ─── 算 Loss ───

為什麼？
  模型不需要「學會」user 的問題長什麼樣——
  它只需要學會：看到這種格式的問題後，該怎麼回答。

  如果也在 user 部分算 Loss，模型會花力氣學習「使用者通常怎麼問問題」，
  這不是我們想要的。
```

**SFT 的規模**

```
Pre-training：~15T Token，幾個月
SFT：         ~10M-100M Token（10 萬到百萬條對話），幾小時到幾天

SFT 的資料量比 Pre-training 小 10 萬倍以上。
但這些少量的高品質資料足以讓模型學會「對話格式」。

為什麼這麼少的資料就夠？
  Pre-training 已經給了模型所有的語言知識。
  SFT 不是在教「新知識」——它是在教「表達方式」。
  就像一個讀了幾億本書的學者，只需要教他「別人問你問題時，
  你應該直接回答而不是開始講故事」。
```

**SFT 資料的品質**

```
高品質 SFT 資料的特徵：
  - 多樣性：涵蓋各種類型的指令（問答、寫作、程式、數學...）
  - 準確性：回答必須正確
  - 格式一致性：嚴格遵循 chat template 的格式
  - 適當的長度和詳細程度

來源：
  - 人工編寫（最貴，品質最高）
  - 用更強的模型生成（便宜，品質中等）
    例如：用 GPT-4 生成 SFT 資料來訓練小模型
  - 開源資料集（如 OpenAssistant、Dolly）
```

**SFT 之後的模型**

```
SFT 前：
  輸入「什麼是機器學習？」
  輸出：「什麼是深度學習？什麼是自然語言處理？...」（續寫模式）

SFT 後：
  輸入「什麼是機器學習？」
  輸出：「機器學習是一種人工智慧技術，它通過從資料中學習來...」（回答模式）✓
```

模型學會了對話的「格式」。但一個新的問題出現了：

```
SFT 後的模型有時候會：
  - 一本正經地胡說八道（Hallucination）
  - 生成有害內容（被 user 用特定 prompt 誘導）
  - 回答品質不穩定（有時很好，有時很差）
  - 太過冗長或太過簡短

SFT 教了模型「怎麼回答」，但沒有教它「什麼是好的回答」。
```

> SFT 是 Pre-training 之後的第一個微調階段——用少量高品質的「指令-回答」對教模型學會對話格式。它改變的是模型的「表達方式」，不是「知識」。但 SFT 只能教格式，不能教品質——這是下一章對齊技術要解決的問題。

---

# 第 8 章：讓模型「說人話」——對齊

## 對齊的目標

SFT 之後，模型會回答問題了，但回答的品質不穩定。對齊（Alignment）的目標是：

**讓模型的回答符合人類的偏好——有幫助、安全、誠實。**

---

## RLHF（Reinforcement Learning from Human Feedback）

RLHF 是最早被廣泛使用的對齊方法（2022 年 ChatGPT 使用的方法）。

### 第一步：收集人類偏好資料

```
給模型同一個問題，讓它生成多個回答（通常 2 個）：

問題：「解釋什麼是黑洞」

回答 A：「黑洞是宇宙中密度極大的天體，它的引力強到連光都無法逃脫。
        當一顆大質量恆星在生命結束時塌縮，就可能形成黑洞。
        黑洞的邊界叫做事件視界...」

回答 B：「黑洞是一個很深很深的洞，所有東西掉進去就出不來了。
        它很黑因為什麼都看不到。科學家用望遠鏡研究它。」

人類標注者判斷：A > B（A 比 B 好）
```

收集大量這樣的偏好對：(question, answer_win, answer_lose)。

### 第二步：訓練 Reward Model（獎勵模型）

```
Reward Model 的結構：
  通常用 SFT 後的模型本身做初始化
  移除 LM Head（不再預測下一個 Token）
  加一個線性層，輸出一個標量分數（reward score）

  輸入：(問題, 回答) → 輸出：一個數字（越高越好）

訓練目標：
  對於每個偏好對 (question, A_win, B_lose)：
  讓 Reward(question, A_win) > Reward(question, B_lose)

損失函數（Bradley-Terry 模型）：
  L = -log(σ(r_win - r_lose))
  σ = sigmoid 函數

  直覺：讓好回答的分數盡可能高於差回答的分數。
```

### 第三步：用 PPO 優化模型

```
PPO（Proximal Policy Optimization）是一種強化學習算法。

在 RLHF 中的角色：
  把 LLM 當作「策略」（Policy）
  把 Reward Model 的分數當作「獎勵」（Reward）
  目標：調整 LLM 的權重，讓生成的回答獲得更高的 Reward

PPO 的訓練循環：

  ┌──────────────────────────────────────────────────┐
  │ 1. 模型生成回答                                    │
  │    問題 → SFT 模型 → 回答                         │
  │                                                    │
  │ 2. Reward Model 打分                               │
  │    (問題, 回答) → Reward Model → 分數 r            │
  │                                                    │
  │ 3. 計算 KL 懲罰                                   │
  │    確保模型不要偏離 SFT 版本太遠                    │
  │    reward_final = r - β × KL(π_new || π_sft)      │
  │                                                    │
  │ 4. PPO 更新                                        │
  │    用 reward_final 作為信號更新模型權重              │
  │    PPO 會限制每步更新的幅度（近端策略）              │
  └──────────────────────────────────────────────────┘
                        │
                        ▼
                  重複幾千到幾萬步
```

### KL 散度懲罰

```
KL(π_new || π_sft) 衡量新模型和 SFT 模型的差異。

為什麼需要它？
  如果不限制，模型會學會「作弊」——
  找到 Reward Model 的漏洞，生成能拿高分但實際上很差的回答。

  例如：Reward Model 可能給長回答高分，
  模型就會生成超長的廢話來「騙分」。

  KL 懲罰確保模型不會偏離 SFT 版本太遠——
  在追求高 Reward 的同時保持語言的自然性。
```

### RLHF 的問題

```
1. 訓練複雜度高
   需要同時維護 4 個模型：
   - 正在訓練的 Policy Model
   - SFT 版本的 Reference Model（算 KL）
   - Reward Model
   - PPO 的 Value Model（估計未來獎勵）

   4 個模型 × 7B 參數 = 28B 參數同時在記憶體中

2. 訓練不穩定
   PPO 的超參數敏感，容易崩潰

3. Reward Model 的偏差
   RM 本身不完美——它也會犯錯，模型可能在學習 RM 的偏差
```

---

## DPO（Direct Preference Optimization）

**為什麼需要它**

2023 年提出的 DPO 解決了 RLHF 的三個問題——不需要 Reward Model、不需要 PPO、訓練穩定。

**核心思想**

DPO 的數學洞察：

```
RLHF 的最終目標是：
  找到一個策略 π，使得
  max E[Reward(x, y)] - β × KL(π || π_ref)

這個最優解有一個解析形式（closed-form solution）：
  π*(y|x) = π_ref(y|x) × exp(r(x,y) / β) / Z(x)

把這個式子反過來，可以用策略的機率表達 Reward：
  r(x,y) = β × log(π(y|x) / π_ref(y|x)) + β × log Z(x)

這意味著：你不需要顯式訓練 Reward Model——
  模型本身的機率分佈隱含了 Reward 資訊。
```

**DPO 的損失函數**

```
L_DPO = -log σ(β × [log π(y_w|x)/π_ref(y_w|x) - log π(y_l|x)/π_ref(y_l|x)])

其中：
  y_w = 人類偏好的回答（winner）
  y_l = 人類不偏好的回答（loser）
  π = 正在訓練的模型
  π_ref = 參考模型（SFT 版本，凍結不動）
  β = 溫度超參數
  σ = sigmoid

直覺：
  log π(y_w|x)/π_ref(y_w|x) = 模型比 SFT 版本更偏好 y_w 的程度
  log π(y_l|x)/π_ref(y_l|x) = 模型比 SFT 版本更偏好 y_l 的程度

  Loss 要求：第一項越大、第二項越小越好。
  也就是：讓模型更偏好好回答，同時更不偏好壞回答。
```

**DPO 的訓練**

```
DPO 的訓練極其簡單：

  輸入：(問題, 好回答, 壞回答) 的三元組

  1. 用模型算 好回答 的 log 機率
  2. 用模型算 壞回答 的 log 機率
  3. 用凍結的 Reference Model 算兩者的 log 機率
  4. 代入 DPO Loss 公式
  5. 反向傳播 + 更新

只需要 2 個模型（訓練模型 + 凍結的參考模型），
不需要 Reward Model，不需要 PPO。

計算量 ≈ RLHF 的 25-60%（取決於配置）。
```

**DPO 的限制**

```
DPO 是離線（Offline）算法：
  訓練資料是固定的偏好對
  模型不會在訓練過程中生成新的回答

  這意味著：模型只能從已有的偏好對中學習，
  無法超越訓練資料的品質上限。
```

---

## GRPO（Group Relative Policy Optimization）

**為什麼需要它**

2024-2025 年，DeepSeek 提出的 GRPO 結合了 RLHF 的「在線學習」和 DPO 的「簡單性」：

```
DPO 的問題：離線，受限於訓練資料的品質
RLHF 的問題：需要 Value Model，訓練複雜

GRPO：在線 + 不需要 Value Model
```

**核心思想**

```
GRPO 的做法：

  1. 給定一個問題 x
  2. 讓模型生成一組回答（Group），比如 G 個：
     y₁, y₂, ..., y_G

  3. 用 Reward 函數（不是 Reward Model）給每個回答打分：
     r₁, r₂, ..., r_G

  4. 計算組內的平均分和標準差：
     μ = mean(r₁...r_G)
     σ = std(r₁...r_G)

  5. 每個回答的 Advantage（相對優勢）：
     A_i = (r_i - μ) / σ

  6. 用 Advantage 更新模型：
     增強 A_i > 0 的回答（比平均好 → 鼓勵）
     抑制 A_i < 0 的回答（比平均差 → 懲罰）
```

**GRPO vs RLHF 的關鍵差異**

```
RLHF 用 Value Model 估計 Advantage：
  需要額外訓練一個模型來預測「未來期望獎勵」
  增加了記憶體和訓練複雜度

GRPO 用 Group 內的相對排名代替 Value Model：
  不需要額外的模型
  用同組回答的平均分作為基準（baseline）
  直覺：好不好是跟同組比的，不是跟一個預測值比的

  這就是「Group Relative」的含義——
  Advantage 是相對於這一組回答的平均水平，不是絕對的。
```

**可驗證獎勵（Verifiable Rewards）**

```
GRPO 的另一個重要創新：不依賴 Reward Model，而是用可驗證的獎勵。

適用場景：數學、程式碼等有明確正確答案的任務

  問題：「計算 23 × 47」
  生成 8 個回答：
    y₁ = "1081" → 正確 → r₁ = 1
    y₂ = "1071" → 錯誤 → r₂ = 0
    y₃ = "1081" → 正確 → r₃ = 1
    ...

  不需要訓練任何 Reward Model——答案對就是對、錯就是錯。

這就是為什麼 2025-2026 年的 reasoning model（如 DeepSeek-R1、
OpenAI o3）大量使用 GRPO + 可驗證獎勵：
  數學、程式碼、邏輯推理 → 答案可以被程式化驗證
  不需要人類標注 → 可以大規模自動訓練
```

---

## 對齊技術的演進

```
時間線：

2022  RLHF + PPO（ChatGPT / InstructGPT）
      首次證明人類偏好對齊能大幅提升模型可用性

2023  DPO（斯坦福）
      去掉 Reward Model 和 PPO，直接從偏好對學習

2024  GRPO（DeepSeek）
      在線 + Group 相對 Advantage + 可驗證獎勵

2025  RLAIF（Claude 系列）
      用 AI 模型代替人類標注偏好

2025  DAPO / GSPO
      GRPO 的改進版本，更穩定的訓練

2026  主流做法：
      SFT → DPO（通用對齊）→ GRPO（推理任務增強）
      或者直接：SFT → GRPO（如果有可驗證獎勵）
```

```
┌───────────────┬───────────────┬───────────────┬───────────────┐
│               │ RLHF          │ DPO           │ GRPO          │
├───────────────┼───────────────┼───────────────┼───────────────┤
│ Reward Model  │ 需要          │ 不需要        │ 不需要        │
│ Value Model   │ 需要          │ 不需要        │ 不需要        │
│ 訓練中生成    │ 是（在線）    │ 否（離線）    │ 是（在線）    │
│ 模型數量      │ 4 個          │ 2 個          │ 2 個          │
│ 訓練穩定性    │ 低            │ 高            │ 中            │
│ 能否超越資料  │ 能            │ 不能          │ 能            │
│ 計算成本      │ 最高          │ 最低          │ 中            │
│ 適合場景      │ 通用對齊      │ 通用對齊      │ 推理任務      │
│ 代表模型      │ ChatGPT       │ Zephyr, Llama │ DeepSeek-R1   │
└───────────────┴───────────────┴───────────────┴───────────────┘
```

> 對齊是 LLM 訓練的第三階段——從 RLHF 的「複雜但有效」，到 DPO 的「簡單且穩定」，到 GRPO 的「在線且可擴展」。2026 年的趨勢是：通用任務用 DPO，推理任務用 GRPO + 可驗證獎勵。對齊不改變模型的知識，而是教它「怎麼表達知識才符合人類期望」。

# 第 9 章：低成本微調——LoRA 與 QLoRA

## 為什麼需要低成本微調

前面的 SFT 和對齊都是**全量微調（Full Fine-Tuning）**——更新模型的所有參數。

```
全量微調的成本（以 7B 模型為例）：
  權重（FP16）：7B × 2 bytes = 14 GB
  梯度（FP16）：14 GB
  優化器狀態（FP32）：7B × 2 × 4 bytes = 56 GB
  Activation（取決於 batch size）：~10 GB
  ──────────────────────────────
  總計：~94 GB → 至少需要 2 張 A100 80GB

  對於 70B 模型：~940 GB → 需要 12+ 張 A100
```

很多場景只需要讓模型「適應一個特定領域」（醫療、法律、金融），不需要改變模型的全部知識。有沒有辦法**只更新一小部分參數**？

---

### LoRA（Low-Rank Adaptation）

**為什麼需要它**

LoRA 的核心假設：**微調時權重的變化量是低秩的**。

什麼意思？

```
Pre-training 學到的權重矩陣 W₀（比如 W_Q）的形狀是 [d × d]。
微調時，我們希望找到一個 ΔW 使得 W_new = W₀ + ΔW。

全量微調：ΔW 也是 [d × d]，有 d² 個參數要學。
  d = 4,096 → d² = 16,777,216 個參數

LoRA 的假設：ΔW 的秩（rank）很低——
  也就是說，ΔW 雖然是 [d × d] 的矩陣，
  但它可以被分解成兩個小矩陣的乘積。
```

**矩陣的秩（Rank）和低秩分解**

```
一個矩陣的秩（Rank）= 它包含的獨立資訊的維度。

直覺：
  一個 [4096 × 4096] 的矩陣有 1600 萬個數字。
  但如果它的秩只有 16，那麼這 1600 萬個數字
  只需要 16 個「基底方向」就能完全描述。

低秩分解：
  如果 rank(ΔW) = r，那麼 ΔW 可以分解為：

  ΔW = B × A

  其中：
    B 的形狀是 [d × r]
    A 的形狀是 [r × d]
    r << d

  驗證維度：[d × r] × [r × d] = [d × d] ✓

以 d = 4096, r = 16 為例：
  原始 ΔW：4096 × 4096 = 16,777,216 個參數
  分解後：  (4096 × 16) + (16 × 4096) = 65,536 + 65,536 = 131,072 個參數

  參數量減少了 128 倍（16M → 131K）。
```

**LoRA 的具體做法**

```
對於每個要微調的線性層（比如 W_Q）：

  原始前向傳播：
    h = W₀ × x      [d × d] × [d × 1] = [d × 1]

  LoRA 前向傳播：
    h = W₀ × x + (B × A) × x
    h = W₀ × x + B × (A × x)
      ↑            ↑
    凍結，不更新   可訓練，r 很小

  計算過程分解：
    步驟 1：A × x → [r × d] × [d × 1] = [r × 1]（壓縮到 r 維）
    步驟 2：B × 結果 → [d × r] × [r × 1] = [d × 1]（投射回 d 維）
    步驟 3：加到原始結果上

  整個過程不需要顯式構造 [d × d] 的 ΔW 矩陣。
```

```
LoRA 的初始化：
  A：用隨機高斯初始化（Kaiming 或正態分佈）
  B：初始化為全零矩陣

  為什麼 B 初始化為零？
  因為 ΔW = B × A，B = 0 → ΔW = 0。
  訓練開始時，模型的行為跟原始 SFT 模型完全一樣。
  訓練過程中，B 逐漸離開零，ΔW 逐漸累積微調的效果。

LoRA 的縮放因子：
  h = W₀ × x + (α/r) × B × A × x

  α = LoRA 的縮放超參數（通常 α = r 或 α = 2r）
  α/r 確保不同的 r 值下，ΔW 的初始學習率量級一致。
```

**LoRA 應用在哪些層？**

```
Transformer Block 中的線性層：

  注意力層：W_Q, W_K, W_V, W_O（各一對 A, B）
  FFN 層：W_up, W_gate, W_down（各一對 A, B）

常見策略：
  - 只對注意力層的 W_Q 和 W_V 加 LoRA（原始論文的推薦）
  - 對所有線性層都加 LoRA（效果通常更好，但可訓練參數更多）

以 Llama 3.1 8B + LoRA rank=16 為例（只對 Q、V）：
  每層可訓練參數：2 × (4096 × 16 + 16 × 4096) = 262,144
  32 層：262,144 × 32 = 8,388,608（≈ 8.4M）
  佔總參數的比例：8.4M / 8B = 0.105%

  只需要訓練 0.1% 的參數！
```

**LoRA 的記憶體節省**

```
全量微調 7B：
  權重 FP16：14 GB
  梯度 FP16：14 GB
  AdamW 狀態 FP32：56 GB
  總計：~94 GB

LoRA 微調 7B（rank=16，只對 Q、V）：
  原始權重 FP16（凍結，只需要存）：14 GB
  LoRA 參數 FP16：8.4M × 2 = ~0.017 GB
  LoRA 梯度 FP16：~0.017 GB
  LoRA AdamW 狀態 FP32：8.4M × 2 × 4 = ~0.067 GB
  Activation（凍結層不需要存梯度的中間結果）：大幅減少
  總計：~20-30 GB → 一張 A100 甚至一張消費級 RTX 4090（24 GB）

記憶體節省 3-4 倍！
```

---

### Rank 的選擇

```
rank r 是 LoRA 最重要的超參數：

r = 4：   參數量極少，適合微小調整
r = 8：   最常用的起始值
r = 16：  常用，大多數任務足夠
r = 32：  較大，接近全量微調的效果
r = 64+： 很少使用，此時不如直接全量微調

r 越大：
  表達能力越強（ΔW 可以捕捉更多變化方向）
  可訓練參數越多
  記憶體需求越大
  過擬合風險越大（如果微調資料少）

r 越小：
  表達能力有限（可能學不到複雜的領域轉換）
  參數量和記憶體最省
  泛化能力通常更好（正則化效果）

經驗法則：
  一般任務（對話風格調整、簡單領域適配）：r = 8 或 16
  困難任務（完全不同的領域、新語言）：r = 32 或 64
```

---

### QLoRA（Quantized LoRA）

**為什麼需要它**

LoRA 已經大幅減少了可訓練參數，但**凍結的原始權重仍然佔用大量記憶體**。7B 模型的 FP16 權重就是 14 GB——一張消費級 GPU（如 RTX 4090，24 GB）勉強放得下。

如果是 70B 模型？FP16 權重 140 GB——根本放不下。

**QLoRA 的解法**：把凍結的原始權重量化到 4-bit。

**量化的基本原理**

```
量化就是用更少的 bit 表示數字：

  FP32（32 bit）：能精確表示 ~7 位有效數字
  FP16（16 bit）：能精確表示 ~3-4 位有效數字
  INT8（8 bit）：只能表示 256 個不同的值
  INT4（4 bit）：只能表示 16 個不同的值

量化的做法（以 absmax 量化為例）：
  1. 找到矩陣中絕對值最大的數 |max|
  2. 把所有數字除以 |max|，映射到 [-1, 1]
  3. 乘以 2^(n-1) - 1，映射到整數範圍
  4. 四捨五入到最近的整數

  FP16 → INT4 的例子：
  原始值：[0.312, -0.876, 0.045, 0.567]
  |max| = 0.876
  正規化：[0.356, -1.000, 0.051, 0.647]
  × 7（4-bit 的範圍是 [-7, 7]）：[2.49, -7.00, 0.36, 4.53]
  四捨五入：[2, -7, 0, 5]

  儲存：只存 [2, -7, 0, 5]（每個數字只需 4 bit）+ |max|=0.876

  反量化（推理時）：
  [2, -7, 0, 5] / 7 × 0.876 = [0.250, -0.876, 0.000, 0.626]
                                vs 原始 [0.312, -0.876, 0.045, 0.567]
  有誤差，但整體方向保持不變。
```

**QLoRA 的三個技巧**

```
1. 4-bit NormalFloat（NF4）量化
   不是均勻量化（0, 1, 2, ..., 15），
   而是按照正態分佈的分位數選擇量化點。
   因為神經網路的權重接近正態分佈，NF4 比均勻 INT4 更精確。

2. 雙重量化（Double Quantization）
   量化需要存一個縮放因子（上面的 |max|）。
   如果每 64 個權重一組，7B 模型就需要 7B/64 ≈ 1.1 億個 FP32 縮放因子
   = 440 MB 的額外開銷。

   雙重量化：把這些縮放因子也量化到 FP8。
   額外開銷從 440 MB → ~55 MB。

3. 分頁優化器（Paged Optimizer）
   AdamW 的 m 和 v 狀態在記憶體不足時自動卸載到 CPU RAM。
   短暫的記憶體峰值（如長序列的 Forward Pass）不會導致 OOM。
```

**QLoRA 的記憶體節省**

```
全量微調 7B：
  ~94 GB → 需要多張 A100

LoRA 微調 7B（FP16 基底）：
  ~20-30 GB → 一張 A100

QLoRA 微調 7B（4-bit 基底）：
  基底權重（NF4）：7B × 0.5 bytes = 3.5 GB
  LoRA 參數（FP16）：~0.017 GB
  LoRA 優化器（FP32）：~0.067 GB
  Activation：~2-5 GB
  總計：~6-10 GB → 一張 RTX 4090（24 GB）綽綽有餘！

甚至 70B 模型：
  基底權重（NF4）：70B × 0.5 = 35 GB
  + LoRA + Activation：~40-50 GB
  → 一張 A100 80GB 就能微調 70B 模型！
```

**QLoRA 的代價**

```
量化引入了精度損失。影響有多大？

  研究結果（2024-2025）：
  - 大多數任務：QLoRA 達到 LoRA 95%+ 的效果
  - 某些對精度敏感的任務（如數學計算）：差距可能更明顯
  - 模型越大，量化的影響越小（因為冗餘更多）

  實務經驗：
  QLoRA 的品質損失通常可以接受。
  如果品質真的不夠，增加 LoRA rank 比換回 FP16 更有效。
```

---

### LoRA 的合併

微調完成後，可以把 LoRA 的權重合併回原始模型：

```
合併公式：
  W_merged = W₀ + (α/r) × B × A

合併後：
  - 模型的結構跟原始模型完全一樣
  - 推理時沒有任何額外開銷
  - 不需要載入 LoRA 的 A 和 B 矩陣

或者，保持 LoRA 未合併：
  - 同一個基底模型可以搭配不同的 LoRA 適配器
  - 切換任務只需要切換 LoRA 權重（幾 MB），不需要換基底模型（幾 GB）
  - 這在多任務部署中非常實用
```

> LoRA 用低秩分解把可訓練參數壓縮到不到原始的 1%；QLoRA 再加上 4-bit 量化，把記憶體需求壓到消費級 GPU 能承受的範圍。它們的核心洞察是：微調不需要改變所有權重——領域適配只需要在少數幾個「方向」上做微調。

# 第 10 章：案例

## 案例 1：追蹤一個 7B 模型的完整訓練過程

讓我們追蹤一個假設的 7B 參數模型從出生到可用的完整旅程。

### 階段 1：Pre-training

```
模型配置：
  參數量：7B
  架構：Decoder-Only Transformer，32 層
  隱藏維度：4,096
  注意力頭數：32
  詞彙表大小：32,000

訓練配置：
  訓練資料：2T Token（過度訓練，D/N ≈ 286）
  Batch Size：4M Token
  序列長度：4,096
  優化器：AdamW（β₁=0.9, β₂=0.95, λ=0.1）
  學習率：Peak 3e-4, Warmup 2000 步, Cosine Decay 到 3e-5
  精度：BF16 混合精度

計算量估算：
  C = 6 × 7B × 2T = 8.4 × 10²² FLOPs
  用 512 張 A100：約 15-20 天
  成本估算：512 × 20 × 24 × $2 ≈ $490K（約 50 萬美元）

訓練過程：
  總步數：2T / 4M = 500,000 步

  Step 0：
    Loss ≈ 10.37（-log(1/32000)，隨機猜的水平）
    模型輸出：完全隨機的 Token

  Step 1,000：
    Loss ≈ 8.5
    模型開始學會：某些 Token 比其他 Token 更常見

    能生成表面流暢但深層邏輯有問題的文字

  Step 300,000：
    Loss ≈ 2.2, PPL ≈ 9

    能正確續寫大部分常見主題的文字

  Step 500,000（結束）：
    Loss ≈ 1.8, PPL ≈ 6
    模型掌握了豐富的語言知識、事實、邏輯推理
    但仍然是「續寫模式」——不會回答問題
```

## 案例 2：用 LoRA 微調一個已有模型

### 場景

你有一個 Pre-trained + SFT + DPO 完成的 7B 模型，想讓它專精於「中醫領域問答」。

```
方案選擇：
  全量微調：~94 GB VRAM → 需要 2 張 A100 → 成本高
  LoRA：    ~25 GB VRAM → 1 張 A100 → 可接受
  QLoRA：   ~8 GB VRAM  → 1 張 RTX 4090 → 消費級 GPU 就行

選擇 QLoRA。
```

### 配置

```
基底模型：7B 參數，NF4 量化 → 3.5 GB
LoRA 配置：
  rank r = 16
  α = 32（α/r = 2）
  應用層：所有 Attention 層（Q, K, V, O）+ FFN（up, gate, down）
  每層 LoRA 參數：7 × (4096 × 16 + 16 × 4096) = 917,504
  32 層：917,504 × 32 = 29,360,128 ≈ 29.4M
  佔總參數：29.4M / 7B = 0.42%
  LoRA 參數記憶體：29.4M × 2 bytes = 58.8 MB

微調資料：
  10K 條中醫問答對
  ≈ 5M Token

訓練配置：
  學習率：2e-4
  Batch Size：4 個序列 × 2048 Token = 8K Token
  Epochs：3
  總步數：5M × 3 / 8K ≈ 1,875 步
```

### 記憶體分析

```
基底模型（NF4）：3.5 GB
LoRA A 和 B（FP16）：0.06 GB
LoRA 梯度（FP16）：0.06 GB
LoRA AdamW 狀態（FP32）：0.24 GB
Activation Cache：~2-4 GB
──────────────────────────
總計：~6-8 GB

一張 RTX 4090（24 GB）綽綽有餘。
訓練時間：~2-4 小時
電費成本：< $5（消費級 GPU）
雲端成本：< $20（租用 GPU）
```

### 微調效果

```
微調前（通用模型）：
  問題：「氣虛體質適合什麼食療？」
  回答：「氣虛體質的人可以嘗試一些滋補的食物。建議諮詢專業醫師。」
  （正確但空泛）

微調後（中醫專精）：
  問題：「氣虛體質適合什麼食療？」
  回答：「氣虛體質以少氣懶言、疲乏無力、自汗為主要表現。
        食療宜選益氣健脾之品：
        - 黃芪燉雞：黃芪 30g、黨參 15g 與母雞同燉
        - 山藥薏仁粥：山藥 30g、薏仁 20g、大米適量
        - 紅棗桂圓茶：紅棗 5 枚、桂圓 10g 泡水代茶
        注意避免生冷、油膩之品，以免損傷脾胃之氣。」
  （專業、具體、符合領域知識）

只用了 0.42% 的參數、$20 的成本，就實現了領域適配。
```

> 案例展示了 LLM 訓練的完整經濟學：Pre-training 花 $490K 建立通用知識（80%+ 的成本），SFT + DPO 花 $100K 教它對話和對齊，而 QLoRA 只需 $20 就能實現領域客製化。

# 第 11 章：收尾

## 壓縮公式

完整版：LLM 訓練是「用海量文本教模型預測下一個 Token（Pre-training）→ 用指令對教它對話（SFT）→ 用人類偏好教它什麼是好回答（Alignment）→ 用低秩矩陣分解低成本客製化（LoRA）」的四階段旅程，每階段都由前一階段暴露的能力缺陷驅動。

口訣版：續寫 → 聽令 → 對齊 → 客製化——每步修補上一步的缺陷。

## 專有名詞速查表

### 損失函數（Loss Function）
衡量模型預測與正確答案之間差距的數字。Loss 越小，預測越準。
- 故事中：訓練的核心目標就是最小化 Loss
- 案例中：從 10.37（隨機猜）降到 1.8（流暢預測）

### 交叉熵損失（Cross-Entropy Loss, CE Loss）
公式 L = -log(p_c)，衡量模型給正確 Token 的機率有多高。梯度 = 預測 - 正確答案。
- 故事中：LLM Pre-training 的標準損失函數
- 案例中：驅動 50 萬步 Pre-training 的優化信號

### 困惑度（Perplexity, PPL）
PPL = exp(Loss)。直覺意義：模型在每個位置平均在幾個 Token 之間猶豫。
- 故事中：Loss 的直覺化版本
- 案例中：從 PPL ≈ 32000（隨機猜）降到 PPL ≈ 6

### 下一個 Token 預測（Next Token Prediction, NTP）
LLM Pre-training 的訓練目標——給定前文，預測下一個 Token。
- 故事中：看似簡單的代理任務，迫使模型學會語言的所有知識
- 案例中：Pre-training 階段每一步的優化目標

### 隨機梯度下降（Stochastic Gradient Descent, SGD）
最基本的優化器：w_new = w_old - η × gradient。所有權重用同一個學習率。
- 故事中：SGD 的三個問題催生了 Adam

### Adam（Adaptive Moment Estimation）
自適應優化器：為每個權重維護一階矩（方向）和二階矩（波動），自動調整學習率。
- 故事中：解決了 SGD 的學習率、噪音、鞍點三個問題

### AdamW（Adam with Decoupled Weight Decay）
Adam 的改進版：把 Weight Decay 從梯度更新中分離出來。LLM Pre-training 的事實標準。
- 故事中：每個參數需要 3 倍記憶體的代價
- 案例中：β₁=0.9, β₂=0.95, λ=0.1 的典型配置

### 學習率排程（Learning Rate Schedule）
訓練過程中學習率的變化規則。Warmup（先升）+ Cosine Decay（再降）是標準配方。
- 故事中：Warmup 防止起步爆炸，Cosine Decay 讓收斂精細化
- 案例中：Peak 3e-4, Warmup 2000 步

### 冪律（Scaling Law）
LLM 的 Loss 與模型大小、資料量、計算量的冪律關係。可以用小實驗預測大模型表現。
- 故事中：揭示了 LLM 能力與規模的數學規律

### Chinchilla 定律
計算最優的模型/資料分配：D_opt ≈ 20 × N。同等計算量下，小模型 + 多資料 > 大模型 + 少資料。
- 故事中：2022 年顛覆了「越大越好」的認知
- 案例中：但實務中模型都在過度訓練（D/N >> 20）

### 過度訓練（Over-training）
有意用遠超 Chinchilla 最優的資料量訓練更小的模型。犧牲訓練效率，換取更低的推理成本。
- 故事中：Llama 3.1 8B 的 D/N ≈ 1,875
- 案例中：7B 模型用 2T Token 訓練（D/N = 286）

### 混合精度訓練（Mixed Precision Training）
Forward/Backward 用 BF16 半精度，權重主副本和優化器狀態用 FP32 全精度。節省記憶體同時保持數值穩定。
- 故事中：讓訓練在有限的 GPU 記憶體中成為可能

### 梯度累積（Gradient Accumulation）
把大 Batch 拆成多個小 Micro-batch，累積梯度後才做一次更新。讓小記憶體也能用大 Batch。
- 故事中：解決「4M Token 的 batch 放不進一台機器」的問題

### 監督式微調（Supervised Fine-Tuning, SFT）
用「指令-回答」對繼續訓練模型，教它從「續寫模式」轉為「對話模式」。只在 assistant 回答部分算 Loss。
- 故事中：Pre-training 後的第一個微調階段
- 案例中：100K 條對話，~1 天，~$1K

### RLHF（Reinforcement Learning from Human Feedback）
用人類偏好訓練 Reward Model，再用 PPO 優化模型。需要 4 個模型同時在記憶體中。
- 故事中：ChatGPT 使用的對齊方法
- 案例中：訓練複雜度高，已逐漸被 DPO/GRPO 取代

### DPO（Direct Preference Optimization）
直接從偏好對學習，不需要 Reward Model 和 PPO。離線算法，計算成本是 RLHF 的 25-60%。
- 故事中：2023 年簡化了對齊的訓練流程
- 案例中：50K 偏好對，~幾小時，~$500

### GRPO（Group Relative Policy Optimization）
在線 RL 方法：生成一組回答，用組內相對排名代替 Value Model。適合有可驗證獎勵的任務。
- 故事中：2024-2025 年推理模型的核心訓練方法

### LoRA（Low-Rank Adaptation）
凍結原始權重，只訓練低秩分解矩陣 ΔW = B × A。可訓練參數 < 1%，記憶體節省 3-4 倍。
- 故事中：核心假設是「微調的權重變化是低秩的」
- 案例中：rank=16，只訓練 0.42% 的參數

### QLoRA（Quantized LoRA）
在 LoRA 基礎上把凍結權重量化到 4-bit（NF4）。7B 模型微調只需 6-8 GB VRAM。
- 故事中：NF4 + 雙重量化 + 分頁優化器三個技巧
- 案例中：$20 成本，一張 RTX 4090 就能微調 7B 模型

### 秩（Rank）
矩陣中獨立資訊的維度數。LoRA 的核心超參數：r 越大表達能力越強但參數越多。
- 故事中：ΔW 的秩遠小於 d → 可以用兩個小矩陣近似
- 案例中：r=16 是常用的起始值
